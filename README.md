# NanoMatch — Ultra-Low Latency Order Matching Engine

A from-scratch C++20 limit order book + matching engine built for the **FEC IIT Guwahati DIY '26 NanoMatch** project. See [NAVIGATION.md](NAVIGATION.md) for a per-file map and [compass_artifact_wf-...md](compass_artifact_wf-6eb4518e-e1eb-4b0b-863f-c00aa39b9544_text_markdown.md) for the design blueprint.

## What it does

```
  ITCH file  -->  mmap (zero-copy)  -->  parser  -->  OrderEvent
                                                          |
                                                          v
                                              single-threaded matcher
                                              (bucket book + slab pool +
                                               intrusive list + O(1)
                                               cancel hash, all L1-resident)
                                                          |
                                                          v   Trade
                                                  SPSC ring buffer
                                                          |
                                                          v
                                                async logger thread
                                                          |
                                                          v
                                                    trades.csv
```

No `malloc`, no `mutex`, no `printf` on the hot path. Strict price-time priority. Integer-tick prices (no floats).

## Build

```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
cmake --build build
```

Optional: enable the SPSC ring + async logger thread (Brick 12) in the demo binary:

```bash
cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release -DNANOMATCH_USE_ASYNC_LOGGER=ON
cmake --build build
```

> On Windows, the async logger uses `std::thread` which trips SmartScreen on unsigned MinGW binaries. The library still compiles it; the demo binary just opts out by default. Turn it on inside WSL2 or after signing the .exe.

## Run

```bash
./build/nanomatch_tests                  # 8 invariants (price-time priority etc.)
./build/nanomatch_bench 2000000          # naive vs optimized, p50/p90/p99 via rdtsc
./build/nanomatch --synthetic 1000000    # end-to-end synthetic, writes trades.csv
./build/nanomatch path/to/S010113-v50.txt  # NASDAQ ITCH 5.0 file replay
```

## Measured results

Generated by `./build/nanomatch_bench 2000000` on Intel i7-14650HX (Raptor Lake-HX, 24 threads, 30 MB L3, TSC 2.419 GHz), MSYS2 UCRT64 g++ 16.1.0, `-O3 -march=native -DNDEBUG`. All per-op timings via **rdtsc** calibrated against `steady_clock`.

| Metric | Naive (Brick 3) | Optimized (Bricks 7-9) | Speedup |
|---|---|---|---|
| Throughput | 5.54 M ops/s | **15.33 M ops/s** | **2.8×** |
| Mean | 160 ns | **43 ns** | 3.7× |
| p50 | 96 ns | **48 ns** | 2× |
| p90 | 384 ns | **48 ns** | 8× |
| p99 | 384 ns | **96 ns** | **4×** |
| p99.9 | 768 ns | **192 ns** | 4× |
| Max | 19.1 ms | **39.6 µs** | **480×** |

The max-latency collapse (19 ms → 40 µs, almost 500×) is the headline figure: it's the cost of `std::map` rebalancing + `_int_malloc` worst case, which the pool + bucket book eliminate entirely.

## Caveats

- Single instrument, in-process. Multi-symbol sharding and network ingest are out of scope.
- `std::unordered_map` is the O(1) cancel index. Swap for `ankerl::unordered_dense` for production-grade numbers.
- Reproduction harness (`scripts/run_bench.sh`: isolcpus, no-turbo, taskset) is Linux-only — use it inside WSL2 for the cleanest p99.
- The blueprint's "Realistic Expectations and Gotchas" section is honest about the laptop-vs-FPGA gap; read it before claiming production parity.
